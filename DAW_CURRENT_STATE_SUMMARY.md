# Current System Summary (2026-01-29)

## Upgrade directions (summary)
- Define a real EffectParamSpec layer (ranges, units, control types, mapping curves) to replace heuristic UI ranges and formatting.
- Remove real-time locks and allocations in render: move FX param updates to lock-free queues or double-buffered snapshots; make graph rebuild thread-safe.
- Add explicit latency reporting/compensation for FX chains and align block processing.
- Unify metering/visualization buses so scopes, meters, and spectrograms share a common capture path.
- Add FX automation lanes and a sample-accurate application path in the render loop.

## Current system overview (detailed)
- Main loop and UI: The application starts by loading config and session/project data, builds UI layout panes, initializes input and inspector/effects panel state, then runs an SDL-driven loop that handles input, updates layout, and renders UI. UI state is centralized and includes effects panel state, layout, selection, and transport UI flags. The UI primarily drives engine actions (transport, FX changes, EQ updates) and reads back meter/spectrum snapshots for visualization.
- Audio engine and threading: The audio device is opened through SDL and registers a callback that simply pulls frames from a lock-free audio queue. A separate worker thread renders audio blocks into that queue. Two additional background threads process spectrum and spectrogram data from ring buffers for UI visualization. The worker thread is the main audio renderer; the audio callback thread only consumes queued audio.
- Audio graph and mixing model: Audio is pull-based. A lightweight graph holds source entries (clips via samplers), and rendering pulls each source into a mix buffer, applies per-clip gain, and accumulates results per track. Track-level processing applies the per-track FX chain, track EQ, pan, and per-track meters before summing into the master buffer. After track summing, master EQ and master FX are applied, the buffer is sanitized for denormals/NaNs, and master meters are updated.
- Transport and timeline: Transport state is a play/stop/seek model. Play/stop/seek are passed to the worker thread via a ring buffer command queue; seek resets the graph and clears the output queue. Looping is supported with start/end frames and is enforced in the worker render loop. There is no distinct pause mode beyond stop.
- Effects system: Effects are managed by a centralized EffectsManager with per-track and master chains stored as arrays of instances. Each effect provides a descriptor (name, param names/defaults, flags) and a vtable for process/set_param/reset/destroy. The manager executes effects in chain order, skipping disabled ones. Meter effects are treated as taps rather than DSP processors, and their audio is forwarded to a meter callback.
- Parameter flow and UI controls: The effects panel is generic and builds its parameter UI from the effect descriptor. Slider ranges and formatting are derived heuristically from parameter names, and tempo-synced parameter modes are managed in the UI layer (beats vs native time/rate). Parameter updates are applied immediately through engine APIs which call into the effects manager; undo snapshots are taken during slider drags.
- EQ detail panel: There is a dedicated EQ curve UI separate from the FX system, with per-track and master curve state stored in UI state. Changes are converted into engine EQ curves and applied via engine commands to update runtime EQ filters.
- Metering and visualization: Track and master meters compute peak/RMS on the worker thread and are stored in shared state under a mutex. FX meters are computed by the meter callback in the render path and are available to the UI via snapshot queries. Spectrum and spectrogram data flow from the worker thread into ring buffers, are processed on background threads, and are read by the UI for visualization views.
- Automation: Automation currently exists only for clip volume and pan. Automation lanes are stored per clip, edited from the timeline/inspector UI, and applied inside the sampler render function via a mutex-protected lane snapshot. There is no FX parameter automation layer.
- Persistence: Sessions are stored in a JSON document that includes engine config, tempo map, timeline view state, track/clip data, automation lanes, and FX chains with parameter values and tempo modes. When loading, FX are staged as pending data and applied after the engine starts.
- Real-time safety status: The audio callback itself is lightweight, but the worker render path uses multiple mutex locks (FX manager, meters, automation). Graph rebuilds and buffer growth can occur without strict synchronization relative to render, and some allocations may happen in the render path during channel changes or mix buffer growth. Logging can occur from the worker thread if enabled.
- Latency handling: Effect descriptors include latency fields, but the engine does not currently report or compensate for latency. Block sizes are fixed by the device configuration.
